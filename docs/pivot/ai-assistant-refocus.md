# AI Assistant Refocus — de Chatbot Genérico a Copiloto de Pricing

> **Parte del plan:** [`PIVOT_PLAN.md`](./PIVOT_PLAN.md) — Bloque C
> **Fecha:** 2026-04-13
> **Estado:** Spec aprobada, implementación pendiente

---

## 1. Motivación

El `AI_LAB` actual usa `@google/genai` (Gemini) como chatbot de propósito general. Con el pivot a motor de pricing avanzado, el AI Assistant debe dejar de ser "haz cualquier pregunta" y pasar a **tres capacidades específicas del workflow pricing**, inyectadas en los puntos donde hoy hay fricción cognitiva para el originator:

1. **Pricing Copilot** — dentro del Calculator, explica por qué el precio recomendado difiere del propuesto comercial.
2. **Loss Classifier** — dentro del Blotter drawer, ayuda a clasificar `loss_reason` desde notas en texto libre.
3. **Negotiation Argument Generator** — dentro del Negotiation Cockpit (post-MVP), genera argumentos de negociación desde contexto del deal.

`AI_LAB` sigue existiendo como playground de exploración, pero las 3 capacidades viven embebidas en su contexto de uso, no en una vista separada.

## 2. Pricing Copilot

### 2.1. Ubicación UX

- Nuevo botón/chip en `CalculatorRecommendationPanel.tsx` (Bloque E): **"Explain this recommendation"**.
- Al clickar, abre slideover con explicación generada.
- Accesible también vía tooltip contextual sobre cada componente del Pricing Receipt.

### 2.2. System prompt

```
You are the N-Pricing Copilot. You help bank originators understand pricing
recommendations generated by the N-Pricing engine.

CONTEXT PROVIDED:
- Deal parameters: amount, currency, tenor, segment, rating, LTV, ESG profile
- FTP breakdown: base rate, liquidity premium, CLC, NSFR, capital charge, ESG
- Proposed rate (commercial input) vs. Recommended rate (EV-optimal)
- Elasticity model inputs: segment, sample size, confidence
- P(win) at proposed vs. recommended rate

YOUR JOB:
1. Explain in plain language (Spanish by default, English if user prompts in English)
   why the recommended rate is X bp away from the proposed rate.
2. Attribute the delta to specific drivers (elasticity signal, capital charge,
   ESG adjustment, liquidity premium) — cite the breakdown numbers verbatim.
3. If the proposed rate would result in significantly lower EV, say so clearly.
4. If the elasticity model confidence is LOW or MEDIUM, flag it.
5. Do NOT invent numbers. Only cite what's in the context.
6. Keep response under 150 words unless user asks for detail.

FORMAT:
- One-sentence verdict (e.g., "Your rate of 4.35% is 17bp above the EV-optimal.")
- 2-4 bullets with driver attribution.
- Final line: confidence qualifier.

REFUSE to give investment advice, credit decisions, or legal guidance.
If asked about non-pricing topics, redirect to the appropriate N-Pricing view.
```

### 2.3. Tool-use (opcional v2)

- `recompute_with_rate(rate: number)` → llama a `calculatePricing` con override y devuelve nuevo `FTPResult`. Permite a Gemini hacer what-if en la respuesta.

### 2.4. Test fixtures

```text
Input: deal Retail, amount 250K€, tenor 60m, ESG green, rating BBB
       proposed=4.35%, recommended=4.18%, P(win) proposed=62%, P(win) rec=78%
Expected output includes:
  - Mentions "17bp" delta explicitly
  - Cites at least ESG greenium or capital charge attribution
  - Mentions P(win) jump
  - Stays under 150 words
```

Tests en `components/Intelligence/__tests__/pricingCopilot.test.ts`.

## 3. Loss Classifier

### 3.1. Ubicación UX

- En Blotter drawer del bloque A: cuando user marca `won_lost = LOST` y tiene el cursor en el campo notes (free-text), aparece sugerencia inline:
  > 💡 AI sugiere: **COMPETITOR** (confianza 80%). [Aceptar] [Descartar]
- Si user acepta, el select `loss_reason` se pre-rellena.

### 3.2. System prompt

```
You are a Loss Reason Classifier for bank deal outcomes.

Given notes written by a bank originator about why a deal was lost, classify
the PRIMARY reason into one of 7 categories:

- PRICE: rate or fee was not competitive enough
- COVENANT: non-price contractual terms (guarantees, reporting, duration flex)
- RELATIONSHIP: broader client relationship influenced the decision
- COMPETITOR: explicit mention of a competitor offer
- TIMING: deal delayed or client urgency mismatch
- CLIENT_WITHDREW: client pulled out for internal (non-pricing) reasons
- OTHER: none of the above fits clearly

OUTPUT FORMAT (strict JSON):
{"classification": "PRICE" | "COVENANT" | ..., "confidence": 0.0-1.0,
 "rationale": "<one sentence>"}

RULES:
- If notes mention "BBVA offered 15bp below" → COMPETITOR.
- If notes mention "no les gustó el covenant" → COVENANT.
- If ambiguous, return OTHER with low confidence.
- Notes may be in Spanish or English.
- Never return free-text categories outside the 7 options.
- If notes are empty or too short, return OTHER with confidence 0.0.
```

### 3.3. Tool signature

```ts
interface LossClassifierRequest {
  notes: string;
  dealContext?: {
    segment: string;
    proposedRate: number;
    competitorRateMentioned: boolean;
  };
}

interface LossClassifierResponse {
  classification: 'PRICE' | 'COVENANT' | 'RELATIONSHIP' | 'COMPETITOR' |
                  'TIMING' | 'CLIENT_WITHDREW' | 'OTHER';
  confidence: number;      // 0-1
  rationale: string;
}
```

### 3.4. Fallback

- Si AI no está disponible (no API key / rate-limit): UI cae a select manual sin sugerencia. Nunca bloquea captura.

### 3.5. Privacy

- Notes pueden contener nombres de clientes y referencias comerciales sensibles.
- Redacción: pre-procesar notes antes de enviarlas a Gemini para strip de nombres propios y números (PII).
- Opción enterprise: usar Gemini vía Vertex AI con `dataProcessing: NO_LOGGING` flag.
- Config flag `VITE_AI_LOSS_CLASSIFIER_ENABLED` para apagarlo en tenants con restricciones.

## 4. Negotiation Argument Generator (post-MVP, Bloque J)

### 4.1. Ubicación UX

- En Negotiation Cockpit (vista futura `/negotiation/:dealId`).
- Panel lateral "Suggested arguments" que se regenera al cambiar counter-offer o contexto.

### 4.2. System prompt

```
You are a Negotiation Argument Assistant for a bank originator negotiating
a loan deal. You generate 3-5 concrete, factual arguments the originator can
use to justify the bank's proposed rate or to negotiate concessions.

CONTEXT PROVIDED:
- Deal parameters including ESG profile, LTV, rating
- Current proposed rate vs. client counter-offer
- Available cross-bonuses (deposits, other products of the client)
- Relationship NPV (ex-post profitability of existing deals with this client)
- Market benchmark rate if available
- Remaining concession budget

YOUR JOB:
Generate arguments grounded in the DEAL CONTEXT. Each argument has:
- A type: "TECHNICAL" | "COMMERCIAL" | "RELATIONSHIP" | "COMPETITIVE"
- A 1-sentence claim
- A quantitative backup (numbers from context)
- A recommended concession (if applicable) with basis-point impact

RULES:
- Never invent numbers. Always cite context.
- If a concession exceeds remaining budget, flag it explicitly.
- If no data supports an argument, skip that argument — do not pad.
- Arguments must be ethically defensible (no misleading statements).
- Spanish by default.

OUTPUT FORMAT:
Array of {type, claim, backup, concession} objects, max 5 items, JSON.
```

### 4.3. Ejemplo de output

```json
[
  {
    "type": "TECHNICAL",
    "claim": "El deal tiene perfil ESG green y cumple taxonomía EU",
    "backup": "ESG greenium aplicable: 15bp según regla activa #E-042",
    "concession": { "rate_bp": -15, "budget_impact_bp": -15 }
  },
  {
    "type": "RELATIONSHIP",
    "claim": "El cliente mantiene depósito de 50M€ a vista con el banco",
    "backup": "Relationship NPV positivo: +240K€ a 3 años",
    "concession": { "rate_bp": -8, "budget_impact_bp": -8 }
  }
]
```

## 9. Riesgos

1. **Latencia de Gemini degrada UX del Calculator.** Mitigación: timeout 8s + loading skeleton + fallback a "Explanation unavailable, see FTP breakdown".
2. **Loss Classifier clasifica mal y sesga el modelo de elasticidad.** Mitigación: confianza < 0.7 → UI no pre-rellena, solo sugiere. User siempre confirma manualmente.
3. **Costes de API Gemini a escala Tier-1.** Retail volumen puede ser alto. Mitigación: caching de respuestas de Pricing Copilot por deal_id (invalidar en cambio de parámetros). Estimación previa requerida antes de go-live.
4. **PII leak a logs.** Mitigación: redacción obligatoria + auditoría en `audit_log` con hash de prompts, nunca texto plano.
